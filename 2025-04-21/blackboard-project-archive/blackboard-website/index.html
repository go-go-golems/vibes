Introduction to Blackboard Systems

Blackboard systems are a type of artificial intelligence architecture that enables collaborative problem-solving by multiple specialized knowledge sources. The blackboard serves as a shared workspace where these knowledge sources can post partial solutions and build upon each other's contributions.

The blackboard metaphor comes from the image of experts standing around a blackboard, working together to solve a problem. Each expert has specialized knowledge and can contribute to the solution when they recognize that their expertise is relevant.

This architecture is particularly useful for problems that:

  • Require diverse types of knowledge
  • Have no clear algorithmic solution
  • Benefit from incremental and opportunistic problem-solving
  • Need to integrate multiple perspectives or expertise

Key Components of Blackboard Systems

The Blackboard

A shared repository of problems, partial solutions, and data that is visible to all knowledge sources.

Knowledge Sources

Independent modules with specialized expertise that can recognize when they can contribute to the solution.

Control Component

Manages the problem-solving process by selecting which knowledge source should be activated next.

Blackboard Architecture

Structure and Organization

The blackboard is typically organized into multiple levels of abstraction or different knowledge areas. Each level represents a different conceptual distance from the raw input data to the final solution.

Knowledge sources are specialized modules that can read from and write to the blackboard. They are triggered when they recognize patterns or conditions on the blackboard that match their expertise.

The control component decides which knowledge source to activate next based on the current state of the blackboard and the potential contributions of each knowledge source. This decision-making process can use various strategies, including:

  • Priority-based selection
  • Focus of attention mechanisms
  • Bidding systems where knowledge sources compete for activation
  • Goal-directed reasoning

Advantages of Blackboard Architecture

  • Modularity: Knowledge sources can be developed independently
  • Flexibility: New knowledge sources can be added without changing existing ones
  • Opportunistic problem-solving: The system can adapt its strategy based on the current state
  • Incremental development: The system can be built and improved gradually
  • Explanation capability: The sequence of knowledge source activations provides a trace of the reasoning process

The Hearsay-II System

Hearsay-II is one of the most influential blackboard systems, developed at Carnegie Mellon University in the 1970s for speech recognition. It pioneered many of the concepts that are now standard in blackboard architectures.

In Hearsay-II, the blackboard was organized into multiple levels of abstraction, from low-level acoustic parameters to high-level semantic interpretations:

  • Parameter Level: Raw acoustic features
  • Segment Level: Phonetic segments
  • Syllable Level: Syllable hypotheses
  • Word Level: Word hypotheses
  • Phrase Level: Phrase structures

Knowledge sources in Hearsay-II included:

  • SEG: Segment Creator (creates segments from parameters)
  • POM: Phone to Syllable (creates syllables from segments)
  • MOW: Syllable to Word (creates words from syllables)
  • PARSE: Syntax Parser (creates phrases from words)
  • PREDICT: Word Predictor (predicts next words based on context)
  • RPOL: Hypothesis Rater (evaluates the credibility of hypotheses)

Hearsay-II Control Mechanism

Hearsay-II used a sophisticated control mechanism based on a focus of attention and a bidding system:

  1. Knowledge sources would "bid" to work on specific parts of the blackboard
  2. Bids included a numeric rating of the expected value of the contribution
  3. The control component would select the highest-rated bid
  4. The focus of attention would shift to the area being worked on
  5. This process would repeat until a solution was found or no more progress could be made

Our Blackboard System Implementation

System Architecture

Our implementation is a modern recreation of the Hearsay-II architecture using contemporary technologies:

  • Backend: Go with SQLite for persistence
  • Frontend: TypeScript with React and Tailwind CSS
  • Communication: REST API, WebSockets, and Server-Sent Events (SSE)

The system follows the classic blackboard architecture with three main components:

  1. Blackboard: A shared data structure that stores hypotheses at different levels of abstraction
  2. Knowledge Sources: Specialized agents that can read from and write to the blackboard
  3. Controller: Manages the activation of knowledge sources based on their bids

Backend Implementation

Database Schema

The SQLite database stores the state of the blackboard, including:

  • Hypotheses at different levels (parameter, segment, syllable, word, phrase)
  • Knowledge sources and their status
  • Activity logs for tracking system behavior
  • Focus control information
  • Bid queue for knowledge source activation

Core Components

The Go backend is organized into several packages:

  • database: Handles SQLite connection and operations
  • models: Defines data structures for the system
  • blackboard: Implements the central blackboard functionality
  • control: Manages the control flow and knowledge source activation
  • agents: Implements the mock knowledge sources
  • api: Provides REST endpoints for frontend communication
  • websocket: Handles real-time updates via WebSockets
  • sse: Implements Server-Sent Events for real-time updates

// Controller implementation
func (c *Controller) cycle() {
    // Get current state
    state := c.bb.GetState()
    
    // Collect bids from knowledge sources
    var bids []bid
    for id, ks := range c.knowledgeSources {
        canActivate, bidValue := ks.CanActivate(state)
        if canActivate {
            bids = append(bids, bid{
                ksID:  id,
                value: bidValue,
            })
        }
    }
    
    // Sort bids by value (highest first)
    sort.Slice(bids, func(i, j int) bool {
        return bids[i].value > bids[j].value
    })
    
    // Activate highest bidding knowledge source
    if len(bids) > 0 {
        highestBid := bids[0]
        ks := c.knowledgeSources[highestBid.ksID]
        
        // Activate the knowledge source
        ks.Activate(c.bb)
    }
    
    // Increment cycle counter
    c.bb.IncrementCycle()
}

Mock Agents Implementation

We implemented six mock agents based on the Hearsay-II knowledge sources:

  1. SEG (Segment Creator): Creates segments from parameters
  2. POM (Phone to Syllable): Creates syllables from segments
  3. MOW (Syllable to Word): Creates words from syllables
  4. PARSE (Syntax Parser): Creates phrases from words
  5. PREDICT (Word Predictor): Predicts next words based on context
  6. RPOL (Hypothesis Rater): Evaluates the credibility of hypotheses

Each agent implements the KnowledgeSource interface:

  • ID(): Returns the agent's identifier
  • CanActivate(): Determines if the agent can contribute and provides a bid value
  • Activate(): Performs the agent's action on the blackboard

// Example agent implementation
func (a *SegmentCreator) Activate(bb *blackboard.Blackboard) error {
    // Get current state
    state := bb.GetState()
    
    // Find a high-confidence parameter to work with
    var selectedParam *models.Hypothesis
    for _, param := range state.Parameter {
        if param.Confidence > 0.9 && !param.Locked {
            selectedParam = ¶m
            break
        }
    }
    
    // Lock the parameter
    bb.LockHypothesis(selectedParam.ID, a.id)
    
    // Create a new segment
    segmentID := fmt.Sprintf("SEG-%03d", a.nextSegmentID)
    a.nextSegmentID++
    
    // Generate segment content
    content := "[DH-AH]"
    
    // Create segment hypothesis
    segment := models.Hypothesis{
        ID:         segmentID,
        Content:    content,
        Confidence: 0.7 + rand.Float64()*0.2,
        TimeRange:  selectedParam.TimeRange + "+100ms",
        Locked:     false,
    }
    
    // Add the hypothesis to the blackboard
    bb.AddHypothesis(segment)
    
    // Unlock the parameter
    bb.UnlockHypothesis(selectedParam.ID)
    
    return nil
}

    // Fetch initial state

      const eventData = JSON.parse(event.data);
      
      // Handle different event types

        case 'hypothesis_added':
        case 'hypothesis_updated':
          // Update state with new hypothesis
          break;
        
        case 'knowledge_source_updated':
          // Update knowledge source
          break;
        
        case 'activity_log_added':
          // Add new activity log entry
          break;
      }
    };
    
    return () => {
      eventSource.close();
    };
  }, []);

  return { state };
};

Applications of Blackboard Systems

Speech Recognition

The original application of the Hearsay-II system. Blackboard architectures are well-suited for speech recognition because they can integrate multiple levels of analysis, from acoustic features to semantic interpretation.

Modern speech recognition systems still use concepts from blackboard architectures, though they are often combined with machine learning approaches.

Image Understanding

Blackboard systems have been used for image understanding tasks, where different knowledge sources can analyze different aspects of an image, from low-level features to high-level semantic content.

Examples include the VISIONS system for scene analysis and medical image interpretation systems.

Intelligent Tutoring

Blackboard architectures have been used in intelligent tutoring systems, where different knowledge sources can represent different aspects of the tutoring process, such as domain knowledge, student modeling, and pedagogical strategies.

The BB1 system is an example of a blackboard-based intelligent tutoring system.

Autonomous Vehicles

Blackboard architectures have been used in autonomous vehicle control systems, where different knowledge sources can handle different aspects of vehicle control, such as perception, planning, and execution.

The NAVLAB system at Carnegie Mellon University used a blackboard architecture for autonomous navigation.

Multi-Agent Systems

Blackboard architectures provide a natural framework for multi-agent systems, where different agents can collaborate by sharing information through a common blackboard.

Modern multi-agent systems often use blackboard-like mechanisms for coordination and communication.

Modern Applications

While pure blackboard architectures are less common today, their concepts have influenced many modern systems:

  • Microservices architectures
  • Event-driven systems
  • Publish-subscribe patterns
  • Complex event processing
  • Hybrid AI systems combining symbolic and neural approaches

Conclusion

Blackboard systems represent a powerful architectural pattern for solving complex problems that require diverse types of knowledge and opportunistic problem-solving strategies. While they originated in the 1970s with systems like Hearsay-II, their concepts continue to influence modern software architecture and artificial intelligence systems.

Our implementation demonstrates how the blackboard architecture can be realized using modern technologies like Go, SQLite, React, and TypeScript. The system successfully recreates the key components of the Hearsay-II architecture, including the multi-level blackboard, specialized knowledge sources, and the control mechanism based on bidding and focus of attention.

As AI continues to evolve, the principles of blackboard systems—modularity, flexibility, and collaborative problem-solving—remain relevant for building systems that can integrate multiple types of knowledge and reasoning approaches.
